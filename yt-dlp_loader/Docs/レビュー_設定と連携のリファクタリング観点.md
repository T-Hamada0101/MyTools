# レビュー: 設定と連携のリファクタリング観点

最終更新日: 2026-04-01

## 1. 目的

この文書は、`yt-dlp_loader` の「設定」と「外部連携」が複数箇所に散らばっている状態を、リファクタリング前提でレビューしたものです。  
単なる感想ではなく、今の実装で実際に起きているズレ、責務の重なり、今後の分離ポイントを明確にすることを目的にします。

## 2. 結論

現状の大きな問題は次の 3 点です。

1. 設定の永続化と画面状態が一致していない
2. `yt-dlp` 実行経路と更新経路が分かれていて責務が揃っていない
3. ブラウザ連携が設定保持、cookie 指定、ブラウザ起動で別々に結び付いている

このため、今のまま機能追加を続けると「設定は保存されたが反映されない」「ボタンごとに動作が違う」「修正箇所が複数に散る」が起きやすいです。  
先に責務境界を切ってから機能追加に入る方が安全です。

## 3. 指摘事項

### [P1] カスタム接頭辞・接尾辞の有効フラグが保存されていない

該当箇所:

- `Form1.cs:27-42`
- `Form1.cs:341-359`
- `YtDlpService.cs:14-30`

内容:

- 画面からは `checkBox7`, `checkBox4`, `checkBox5` を読んで `UseCustomPrefix1`, `UseCustomSuffix1`, `UseCustomSuffix2` を組み立てています
- しかし設定ロード時は対応するチェック状態を復元していません
- さらに `SaveSettings` でも有効フラグ自体を保存していません

影響:

- 文字列だけ保存され、次回起動時に「値は残るが有効にならない」状態になります
- 画面上の見た目と `yt-dlp.conf` の生成条件が一致しません

リファクタリング方針:

- 永続化対象を `AppSettings` として明示し、文字列値と有効フラグを同じモデルで保存する
- `LoadProperties` と `SaveSettings` を 1 対 1 で対応させる

### [P1] 設定項目と実際の実行設定が未接続な箇所がある

該当箇所:

- `Form1.cs:333-349`
- `YtDlpService.cs:131-180`

内容:

- `DLThreads` は画面から読み取り保存しています
- しかし `BuildConfigLines` ではスレッド数に相当する設定を一切出力していません

影響:

- ユーザーは設定したつもりでも実行結果に反映されません
- 設定項目の信頼性が落ち、リファクタリング時の正解データも曖昧になります

リファクタリング方針:

- 「保存だけする設定」と「実行に使う設定」を分ける
- 未接続の項目は、接続するか削除するかを先に決める

### [P1] `yt-dlp` 実行経路が複数あり、設定連携が分岐している

該当箇所:

- `Form1.cs:145-191`
- `Form1.cs:309-324`
- `Form1.cs:394-397`
- `YtDlpService.cs:44-115`
- `yt-dlp.cs:14-92`

内容:

- 通常実行は `YtDlpService.RunYtDlp` を通ります
- `1 movie DL` は `Process.Start` を直接呼び、`yt-dlp.conf` も `YtDlpService` も通りません
- `Update` は別クラス `yt_dlp.UpdateYtDlp()` を使い、実行系の責務が分かれています

影響:

- ボタンごとに設定反映のされ方が異なります
- 実行引数、ログ取得、例外処理、更新処理を共通化しにくいです
- リファクタリング時に「どれが正式な実行経路か」が曖昧です

リファクタリング方針:

- `YtDlpRunner` のような単一の実行窓口を作る
- 通常実行、単体実行、更新実行を同じ経路で扱えるようにする
- `yt-dlp.cs` は役割を吸収して削除候補にする

### [P2] 保存済み設定と画面入力のどちらを使うかがボタンごとに揺れている

該当箇所:

- `Form1.cs:284-287`
- `Form1.cs:299-301`
- `Form1.cs:394-397`

内容:

- `Show DL Folder` は `Settings.Default.DownloadDirectory` を直接使います
- `Update` は `Settings.Default.ExePath` を使います
- 一方で `Start` は画面から `BuildOptionsFromControls()` で値を集めます

影響:

- 画面上で値を変えても、保存前に押したボタンでは古い設定が使われることがあります
- ユーザー視点では「同じ画面なのにボタンごとに参照元が違う」状態です

リファクタリング方針:

- UI 操作はまず `CurrentFormState` を組み立てる
- 各ボタンは必ずその状態を使う
- 保存が必要な操作だけ `SettingsStore` を呼ぶ

### [P2] ブラウザ連携の責務が 1 つのクラスに混在している

該当箇所:

- `Browser.cs:15-18`
- `Browser.cs:38-76`
- `Browser.cs:122-160`
- `Form1.cs:22`
- `Form1.cs:208-210`
- `Form1.cs:241-248`

内容:

- `Browser.cs` は JSON 読み込み、既定値決定、cookie オプション生成、ブラウザ起動引数生成をまとめて持っています
- UI のコンボボックス変更時には、その場で設定ファイルを書き換えています

影響:

- 設定読み込みと実際の起動処理が強く結び付きます
- 将来ブラウザ追加や起動方式変更をすると、設定周りにも影響が出やすいです

リファクタリング方針:

- `BrowserProfileRepository`
- `BrowserCookieOptionBuilder`
- `BrowserLauncher`

上の 3 役に分けると見通しがよくなります。

### [P2] ブラウザ対応の宣言と実装が揃っていない

該当箇所:

- `Browser.cs:148-158`
- `Browser.cs:163-198`
- `browser_profiles.json`

内容:

- `SupportedBrowsers` と `GetBrowserExecutable` では `chromium`, `edge`, `brave`, `opera` まで列挙しています
- しかし実際の `CreateProcessStartInfo` は `chrome` とそれ以外を `firefox` として扱っています
- 実行ファイルのパスもコードに固定です

影響:

- JSON 上は拡張できそうに見えても、実際には起動できない組み合わせがあります
- 設定ファイル上の表現力と実装の実力がずれています

リファクタリング方針:

- サポート対象をまず明示的に絞る
- そのうえで、対応ブラウザごとに起動戦略を分ける
- 未対応ブラウザは早い段階で検証エラーにする

### [P2] `Form1` がオーケストレーションとインフラ詳細を持ちすぎている

該当箇所:

- `Form1.cs:27-58`
- `Form1.cs:145-191`
- `Form1.cs:231-263`
- `Form1.cs:331-372`

内容:

- 画面イベントの中で設定組み立て、保存、設定ファイル出力、プロセス実行、ブラウザ起動、ログ反映まで抱えています

影響:

- テストしにくいです
- UI 変更と実行処理変更が同じファイルに集中します
- リファクタリング時に副作用の影響範囲が読みにくいです

リファクタリング方針:

- `Form1` は入力と表示に寄せる
- 実処理は `ApplicationService` 層へ逃がす

### [P3] 設定定義が複数ファイルに分散し、差分が出ている

該当箇所:

- `Properties/Settings.settings`
- `Properties/Settings.Designer.cs`
- `App.config`

内容:

- `Settings.settings` と `Settings.Designer.cs` には `AddPrefix1` があります
- しかし `App.config` には同名設定が見当たりません

影響:

- 設定追加時の変更点が複数に散り、追従漏れが起きやすいです
- 実行時影響は構成次第ですが、少なくとも保守上のノイズです

リファクタリング方針:

- 設定定義の正本を 1 つに寄せる
- 自動生成物と手書き設定の責務を混ぜない

### [P3] 失敗時の扱いが弱く、静かに無視する経路がある

該当箇所:

- `YtDlpService.cs:35-38`
- `YtDlpService.cs:46-49`
- `Browser.cs:70-73`
- `Form1.cs:365-368`

内容:

- 空パスや JSON 読み込み失敗の一部が、そのまま黙って返る作りです
- UI 側では失敗理由が出ません

影響:

- 不具合解析が UI 操作に依存します
- 分離後のサービス化で原因追跡しにくくなります

リファクタリング方針:

- `Result` か例外方針を決める
- 失敗理由を UI に返せる形へ統一する

## 4. 目指す構成

設定と連携を次の単位に分けると整理しやすいです。

### 4.1 設定

- `AppSettings`
  画面で保持したい永続化対象
- `AppSettingsStore`
  `Settings.Default` との変換を担当
- `FormStateMapper`
  画面 <-> `AppSettings` の変換を担当

### 4.2 yt-dlp 実行

- `YtDlpRequest`
  実行時だけ必要な情報
- `YtDlpConfigBuilder`
  `yt-dlp.conf` の文字列生成
- `YtDlpRunner`
  実行、更新、単体実行を一元化

### 4.3 ブラウザ連携

- `BrowserProfileRepository`
  `browser_profiles.json` の読込
- `BrowserLauncher`
  URL を開く責務
- `BrowserCookieOptionBuilder`
  `--cookies-from-browser` の生成

## 5. 最初にやるとよい順番

1. `AppSettings` を作り、保存対象を明示する
2. `LoadProperties` と `SaveSettings` の対応漏れを解消する
3. `buttonStart_Click` から実処理を `ApplicationService` へ移す
4. `yt-dlp.cs` を整理し、実行窓口を 1 本にする
5. `Browser.cs` を repository / launcher / option builder に分割する
6. 未接続設定の扱いを決める

## 6. 判断

このリポジトリは、全面作り直しより「設定モデルの一本化」と「実行窓口の一本化」から入るのがよいです。  
特に次の 2 点は先に直す価値が高いです。

1. カスタム接頭辞・接尾辞の有効フラグを保存対象に入れる
2. `Start` / `Update` / `1 movie DL` の実行経路を共通化する

ここを揃えるだけでも、設定と連携の散らばりはかなり減ります。
